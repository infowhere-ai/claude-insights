## Atividades Detalhadas

### 1. Fix false WORKING state após sessão terminar
**O que foi feito**:
- O `jsonl_watcher_loop` e o `poll_loop` estavam a forçar o estado `working` sempre que o JSONL file tinha menos de 60s, mesmo quando o hook `Stop` já tinha escrito `idle` no `status.json`
- Fix: adicionada condição `jsonl_mtime > status_mtime` em ambos os loops — o JSONL só sobrepõe o `status.json` se for mais recente (ou seja, se os hooks não capturaram o evento)
- Se `status.json` for igual ou mais recente (Stop hook correu), o estado dos hooks é respeitado

**Arquivos modificados**:
- `app.py` — condição `jsonl_mtime > mtime` em `poll_loop` (linha ~293) e `latest_mtime > status_mtime` em `jsonl_watcher_loop` (linha ~393)

**Resultado**: Sessões terminadas passam a idle correctamente sem esperar 60s

### 2. Fix notificação "Claude needs your permission" persistindo após resposta
**O que foi feito**:
- Quando o hook `Notification` escrevia `notification: {...}` e de seguida o JSONL watcher sobrepunha o estado para `working`, copiava o `current` sem limpar o campo `notification`
- A notificação "Claude needs your permission to use Bash" continuava a aparecer mesmo depois do utilizador responder e do `PreToolUse` ter disparado
- Fix: ao forçar estado `working` via JSONL (em `jsonl_watcher_loop` e `poll_loop`), limpar explicitamente `notification = None`

**Arquivos modificados**:
- `app.py` — `updated["notification"] = None` e `data["notification"] = None` nas duas sobreposições JSONL

**Resultado**: A notificação de permissão desaparece assim que o JSONL confirma que Claude está a executar

### 3. Sistema de pin para controlo do monitor
**O que foi feito**:
- Bug: `project-finances` aparecia no monitor principal sem ter pin (alfinete)
- O monitor mostrava todos os projectos em `allProjects` (sessões activas) independentemente de estarem pinados
- Fix: `renderAll()` passou a filtrar `visible` por `pinnedProjects.has(name)` — só projectos pinados aparecem no monitor
- Fallback: se não há nenhum pin configurado (`pinnedProjects.size === 0`), mostra todos (compatibilidade inicial)
- O painel ALL PROJECTS continua a mostrar todos os projectos (ACTIVE = tem sessão, PENDING = sem sessão), independentemente do pin

**Arquivos modificados**:
- `static/index.html` — filtro `(!hasPins || pinnedProjects.has(name))` em `renderAll()`

**Resultado**: Apenas projectos com alfinete aparecem no monitor; despinar move o projecto para fora do monitor; o painel ALL PROJECTS mantém visibilidade total

### 4. Redução de delay working/idle
**O que foi feito**:
- Working latch no frontend: quando transitava de `working → idle`, o frontend mantinha `working` visível por até 3s (anti-flash entre ferramentas consecutivas) — delay muito perceptível no final de sessão
- Poll interval: 1.0s significava até 1s de atraso na deteção de mudanças no `status.json`
- Fix: latch reduzido de 3000ms para 800ms; poll interval reduzido de 1.0s para 0.5s

**Arquivos modificados**:
- `static/index.html` — `WORKING_LATCH_MS = 800`
- `app.py` — `POLL_INTERVAL` default de `"1.0"` para `"0.5"`

**Resultado**:
- `idle → working`: até ~500ms (antes: ~1000ms)
- `working → idle`: até ~1.3s (antes: ~4s)

### 5. Fix Context tab — filtro de projetos e contagem de agentes
**O que foi feito**:
- O Context tab mostrava abas de projectos que não estavam pinados (ex: `standarts-llm`, `copilot-meeting`)
- Fix: substituído o filtro por tokens/modelo por `!hasPins || pinnedProjects.has(name)` — mesma lógica que o monitor principal
- Problema adicional: o Context mostrava "7 agents" (total cumulativo da sessão via `stats.sub_agents_total`) enquanto o card mostrava "3 running" (agentes ativos via `active_agents`)
- Fix: Context passou a calcular `activeAgents.filter(a => a.state === 'running').length` a partir de `data.active_agents`

**Arquivos modificados**:
- `static/index.html` — filtro de abas no `renderContextTab()` e cálculo de agentes ativos

**Resultado**: Context tab mostra só projetos pinados; contagem de agentes coincide com o card

### 6. Fix agent modal — onclick quebrado por JSON.stringify
**O que foi feito**:
- O botão de ferramenta nos sub-agentes (`▶ Read`) não abria o modal ao clicar
- Causa: `onclick="openAgentModal(${JSON.stringify(storeKey)})"` gerava `"project::key"` com aspas duplas dentro de atributo HTML com aspas duplas — atributo inválido, onclick nunca disparava
- Fix: substituído por `data-agent-key="${escAttr(storeKey)}"` + `addEventListener('click', ...)` via JS após o `innerHTML`

**Arquivos modificados**:
- `static/index.html` — `renderAgents()`: remover inline onclick, usar data-attribute + event listener

**Resultado**: Clicar no `▶ Read` / `▶ Bash` abre o modal com detalhes do agente
