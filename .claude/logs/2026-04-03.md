## Atividades Detalhadas

### 1. Auditoria completa da aplicação (read-only)

**O que foi feito**:
- Leitura completa de `app.py` (1145 linhas) e secções relevantes de `static/index.html`
- Medição de timings reais: SSE connection (35.7ms), primeiro init (240ms), cadência de updates (~2s), `/api/pending` (22-42ms), `/api/account` (3-4 segundos a bloquear)
- Identificados todos os problemas com root cause e localização exacta no código

**Problemas encontrados**:
- ❌ CRÍTICO: `/api/account` lê 1708 ficheiros JSONL (648MB) de forma síncrona num handler `async` — bloqueia event loop ~3.7s; nenhum SSE event entregue durante esse período
- ⚠️ BUG: variável `now_ts` sobrescrita dentro do loop de agents em `poll_loop` (linha 327) — shadow da variável outer do mesmo ciclo
- ⚠️ PERFORMANCE: `loadPendingFiles()` disparado por projeto a cada SSE `update` — com 4 projectos = 4 HTTP calls a cada 2s
- ℹ️ MINOR: flag `_stale = True` definido em `jsonl_watcher_loop` mas nunca lido pelo frontend (dead code)

**Resultado**: panorama completo entregue ao utilizador com medições reais, root causes e prioridade de fixes

### 2. Fixes aplicados

**O que foi feito**:

- **Fix 1 — `/api/account` desbloqueia event loop**: extraída toda a lógica de scan para `_get_account_sync()` (função síncrona). O handler passou a `await asyncio.to_thread(_get_account_sync)`. O scan de 1708 ficheiros corre numa thread separada — o event loop fica completamente livre durante os ~3-4s da operação.

- **Fix 2 — `now_ts` shadowing**: a variável interna ao loop de agents em `poll_loop:327` foi renomeada de `now_ts` para `agents_now`. A referência na comparação de idade dos agents foi actualizada para `agents_now`.

- **Fix 3 — `loadPendingFiles` debounce**: adicionado `loadPendingFilesDebounced(projectName)` com timer de 3s por projecto. O SSE handler de `update` passou a usar a versão debounced. Chamadas rápidas sucessivas coalescem numa única fetch. O `loadAllPendingFiles` a cada 10s e o handler `init` continuam com a versão imediata.

**Arquivos modificados**:
- `app.py` — extraída lógica de `/api/account` para `_get_account_sync()`, handler usa `asyncio.to_thread()`; `now_ts` → `agents_now` no loop de agents
- `static/index.html` — adicionado `loadPendingFilesDebounced()` com `_pendingDebounceTimers`; SSE `update` handler usa versão debounced

**Resultado**: event loop deixou de bloquear na abertura do tab Account; HTTP calls de pending files reduzidas de N/update para 1 a cada 3s mínimo por projecto

### 3. Investigação crítica da Context tab + 3 fixes

**O que foi feito**:

- **Investigação live**: monitoramento do SSE stream e comparação directa entre JSONL e status.json enquanto sessões estavam activas. Confirmado que os valores chegam correctamente ao backend; o problema estava em dois pontos do merge e num bug de UI.

- **Bug identificado 1 — nova sessão zera ctx**: Quando uma nova sessão JSONL começa sem mensagens `assistant` ainda, `_get_project_stats()` retorna `session_ctx_tokens=0`. Em `jsonl_watcher_loop`, a ordem `{**hook_stats, **jsonl_stats}` (JSONL ganha) fazia o valor 0 sobrescrever o valor válido do hook — ctx% desaparecia da Context tab.

- **Fix 1 — guard anti-zero**: Após o merge em `jsonl_watcher_loop` (linhas 459-462 e 487-491), adicionado guard explícito: se JSONL retorna `session_ctx_tokens=0` mas hook tem valor não-zero, preserva o valor do hook. Mesmo guard para `model=""`. Aplicado nas 2 ocorrências dentro do loop (sessão activa e sessão com sub-agents).

- **Bug identificado 2 — ctxSelectedProject desincronizado com pin filter**: Se o utilizador tinha um projecto seleccionado na Context tab e depois activava pins que não incluíam esse projecto, o `sorted` (lista de tabs) só mostrava projectos pinados, mas o painel de detalhe ainda mostrava o projecto não-pinado. Nenhuma tab ficava marcada como activa.

- **Fix 2 — reset ctxSelectedProject contra sortedNames**: Substituída a condição `!allProjects[ctxSelectedProject]` (que nunca era true porque o projecto existe em `allProjects` mesmo sem pin) por `!sortedNames.has(ctxSelectedProject)` onde `sortedNames` é o Set dos projectos visíveis após filtro.

- **Restart do servidor**: Servidor estava a correr com o código pré-fix (PID 88184 desde antes das correcções). Killed e reiniciado com novo PID 45677 com todas as correcções activas.

- **Validação live**: Monitorização do polling a cada 2s por 16s — ctx estável, sem oscilação, a incrementar correctamente (146597 → 148218 → 148598) durante trabalho activo.

**Arquivos modificados**:
- `app.py` — guard anti-zero em `jsonl_watcher_loop`: 2 pontos de merge protegidos (sessão activa + sessão com sub-agents)
- `static/index.html` — fix `ctxSelectedProject` vs pin filter: usa `sortedNames.has()` em vez de `allProjects[name]`

**Resultado**: Context tab actualiza correctamente — ctx nunca vai a zero no início de nova sessão, sem oscilação entre loops, tab seleccionada sempre visível na lista filtrada

### 4. Fix do estado "compacting" sobrescrito por "working"

**O que foi feito**:

- **Investigação**: Diagnosticado por que o monitor não mostrava nenhuma informação durante a compactação da janela de contexto. O `PreCompact` hook escreve `state="compacting"` no `status.json` (mtime=T1). O novo JSONL criado durante a compactação tem mtime=T2 > T1. Ambos os loops interpretam isso como "sessão activa" e substituem o estado por "working" — o frontend nunca vê "compacting".

- **Fix 1 — `poll_loop` (linhas 306-313)**: Adicionada guarda `if data.get("state") != "compacting":` antes de sobrescrever o estado para "working" e antes de limpar a notificação. Enquanto o estado for "compacting", o loop não interfere.

- **Fix 2 — `jsonl_watcher_loop` (linhas 436-444)**: Adicionada guarda `if cur_state != "compacting":` para proteger o `updated["state"] = "working"`. A limpeza da notificação também foi guardada com `and cur_state != "compacting"` — preserva a mensagem "⚡ Compactando contexto" definida pelo hook durante toda a compactação.

- **Por que funciona**: A transição de volta a "idle" só ocorre quando `cur_state == "working"` (linha 488). Como o estado nunca passa para "working" durante a compactação, essa transição também fica bloqueada. O estado "compacting" persiste até o primeiro hook da nova sessão escrever um estado novo.

**Arquivos modificados**:
- `app.py` — guarda anti-override em `poll_loop` e `jsonl_watcher_loop`; notificação de compactação preservada

**Resultado**: Monitor mantém estado "compacting" visível durante toda a compactação; servidor reiniciado com PID 52122
